# 캡차 위젯(랜덤 미션 + 시간제한) 검증 체크리스트

**작성일**: 2026-06-17
**대상**: 실제 운영 중인 캡차 위젯(별도 레포, ML팀은 코드 접근 불 가)을 만든/관리하는 팀
**목적**: ML팀이 가진 `captcha_engine_team_handoff`(레퍼런스 데모 코드)와 실제 위젯이 보안상 핵심적인 부분에서 어떻게 다른지 확인. 답변에 따라 replay/인젝션 공격에 대한 시스템 전체 위험도 평가가 크게 달라짐.

> 배경: ML팀이 가진 모델(v3 + source-weight×8, threshold≈0.65)은 단독으로 replay 공격을 약 45~55% 통과시킨다. 이게 실제 서비스에서 얼마나 위험한지는 **미션·시간제한이 이 빈틈을 얼마나 메워주는지**에 달려 있는데, 그건 위젯 구현에 달려 있어 ML팀이 가진 코드만으로는 답할 수 없다.

## ⭐ 2026-06-17 업데이트 — 실제 미션 카탈로그 확인됨

서비스 담당자로부터 실제 운영 중인 랜덤 미션 정의를 직접 전달받았다. **A1, C5는 이걸로 확인 완료** — 더 이상 추측이 아니다.

**얼굴 미션** (약 10종): 미소 짓기 / 왼쪽 눈 감기 / 오른쪽 눈 감기 / 고개 끄덕이기 / 도리도리하기 / 입벌리기 / 고개 상·하·좌·우 돌리기(4방향)

**손동작 미션** (약 7종 + 손가락 번호는 조합형): 브이하기 / 따봉하기 / 동그라미(OK사인) / 손가락 번호 지정(예: 2·5번만 펴기 — 5개 중 2개 고르는 조합이라 그 자체로 다수 변형) / 손 흔들기 / 손을 좌·우로 밀기 / **손가락으로 좌·우 볼 찌르기**

→ ML팀 데모(`hand_gesture_detector.py`)의 4종(index_up/two_fingers/open_palm/fist)보다 훨씬 풍부하다. C(엔트로피) 우려는 상당 부분 해소됐다고 봐도 된다 — 최소 10(얼굴) × 10~15(손, 손가락조합 포함) 조합이면 정적 공격자가 클립 몇 개 준비해서 대응하기 어렵다.

**B(손-얼굴 결합)에 대한 부분적 희소식, 그러나 전부는 아님**: "손가락으로 볼 찌르기"는 **손이 물리적으로 얼굴에 닿아야 하는 유일한 미션**이다. 이게 제대로 구현되어 있다면(손가락 끝 landmark와 볼 landmark의 거리를 실제로 검사한다면) 이 미션이 뽑혔을 때만큼은 "재생된 얼굴 + 라이브 손" 공격이 막힌다.
**단, 나머지 6종 손동작(브이/따봉/OK/번호/흔들기/밀기)은 얼굴과의 공간적 연관이 전혀 없다** — 이 6종 중 하나가 뽑히면 B의 취약점이 그대로 남는다. 즉 "볼 찌르기"가 뽑힐 확률(대략 1/7 × 손 라운드가 나올 확률)만큼만 방어력이 생긴다 — 구조적으로 풀린 게 아니라 일부 라운드에서만 우연히 막히는 수준.

→ 그래서 G3 테스트(진짜 손으로 제스처 + 얼굴은 재생영상)는 **반드시 "볼 찌르기"가 아닌 다른 미션(예: 브이하기)으로** 시도해봐야 한다. "볼 찌르기"로 테스트하면 우연히 막혀서 B 전체가 안전하다고 오판할 수 있다.

## ⭐⭐ 2026-06-17 업데이트 2 — A2 확인됨: 미션은 spoof_score와 "완전 독립적"

서비스 담당자 확답: 미션 pass/fail 판정이 spoof_score 재탕이 아니라 **지시받은 동작을 실제로 검증**한다. 체크리스트 전체에서 가장 무겁다고 했던 질문이 "양호"로 해소됐다.

**이게 바꾸는 결론**:
- 단순 고정 영상 재생(공격자 없이 화면만 틀어놓는 가장 흔한 자동화 공격)은 17종 미션 중 랜덤하게 뜨는 지시에 그 순간 반응할 수 없으므로 **시스템 레벨에서 모델 단독 수치(45~55% 통과)보다 훨씬 강하게 막힐 가능성이 높다.** 즉 "55% 통과는 시스템 위험도를 과대평가한 것"이라는 원래 가설이 (적어도 단순 정적 공격에 대해서는) 맞는 쪽으로 기울었다.
- 남는 잔여 위험 둘:
  1. **클립 교체형 공격**: 17종 각각에 대응하는 클립을 미리 준비해 지시에 맞춰 바꿔 끼우는 공격자. 비용은 크게 올라가지만 이론상 가능.
  2. **B의 잔여 취약점(위 업데이트 1 참고)**: "볼 찌르기" 외 6종 손동작은 얼굴-손 결합 검증이 없을 가능성 — 라이브 손 + 재생 얼굴 조합 공격에는 미션 다양성이 도움이 안 됨. A2가 좋아진 것과 B는 별개 축이다.
  3. 실시간 인젝션/딥페이크는 여전히 미션 다양성을 무력화함(E 항목, 변동 없음).

→ 전체적으로 **배포 가능성 판단이 한 단계 긍정적으로 이동**했다. 다만 B(특히 G3 테스트 결과)와 E(인젝션 방어)가 아직 미확인이라 최종 결론은 보류.

## A. 얼굴 미션의 실체

1. ~~얼굴 미션이 다양하고 랜덤한가~~ → **확인됨, A1 해소**
2. ~~얼굴 미션의 pass/fail이 spoof_score와 독립적인가~~ → **확인됨 (2026-06-17, 서비스 담당자 확답): "완전 독립적"이다.** 즉 mission_pass는 spoof_score 재탕이 아니라, 지시받은 동작(왼쪽 눈 감기 등)을 실제로 했는지 별도 검증한다. ML팀 데모 코드(`mission_pass = spoof_score <= threshold`)와는 다른 구조 — **A2는 해소, "양호"로 확정.**

   **이게 바꾸는 결론**: 미션 카탈로그(17종)가 진짜로 두 번째 방어선 역할을 한다. 사전 녹화 영상 1개로는 랜덤하게 뜨는 지시("지금 도리도리 해주세요")에 그 순간 반응할 수 없으므로, **단순 고정 영상 재생 공격은 모델 단독 수치(45~55% 통과)보다 시스템 레벨에서 훨씬 강하게 막힐 가능성이 높다.** 다만 두 가지 잔여 위험은 남는다:
   - **클립 교체형 공격**: 공격자가 17종 미션 각각에 대응하는 사전 녹화 클립을 미리 만들어두고, 뜨는 지시에 맞춰 클립을 바꿔 끼우는 경우. 비용은 올라가지만 불가능하진 않음.
   - **실시간 인젝션/딥페이크**: 라이브로 지시에 반응 가능하므로 미션 다양성 자체가 무력화됨. (E 항목, 별도 과제)

## B. 손-얼굴 결합 여부 (가장 중요, 위 업데이트로 일부만 해소)

3. "손가락으로 볼 찌르기" 미션의 pass 판정이 **실제로 손가락 끝과 볼의 거리(occlusion/접촉)를 검사**하는가, 아니면 이때도 "손가락이 펴져 있나"만 보고 위치는 안 보는가?
4. 나머지 6종 손동작(얼굴과 공간적 연관 없는 것들)이 뽑혔을 때: 공격자가 **얼굴은 재생 영상으로 가리고, 자기 진짜 손으로 그 제스처만 수행**하는 시나리오를 막을 방법이 있는가?
   - (ML팀이 가진 레퍼런스 데모는 손과 얼굴을 같은 프레임에서 각자 독립적으로만 검사함 — 결합 검증 없음. 실제 위젯도 "볼 찌르기" 외 미션에서는 같은 구조일 가능성.)

## C. 손 미션의 다양성

5. ~~지원되는 손 제스처 종류가 몇 개인가~~ → **확인됨 (위 업데이트 참고), 7종+조합형 — 충분히 풍부함**
6. (남은 질문) 같은 사용자가 캡차를 여러 번 시도할 때 미션이 매번 새로 랜덤 추첨되는가, 아니면 세션 내내 같은 미션이 반복되는가? — 추첨 자체가 매번 일어나는지는 별개 확인 필요

## D. 라운드 간 독립성

## G. 코드 안 봐도 되는 블랙박스 테스트 (실제로 위젯을 조작해서 결과만 확인)

코드를 읽고 설계 의도를 보고하는 것보다, **아래 테스트를 실제로 수행해서 통과/실패 결과만 보고**하는 게 더 빠르고 정확하다. 필요한 준비물: 캡차 위젯이 떠 있는 브라우저, 개발자도구(Network 탭), 동영상 1개(자기 자신이 가만히 정면을 보고 있는 영상이면 충분), 가능하면 OBS Virtual Camera 같은 가상 카메라 소프트웨어.

| # | 테스트 방법 | 무엇을 확인하는가 | 결과 해석 |
|---|---|---|---|
| G1 | 얼굴 미션 지시(예: "왼쪽 눈 깜빡여주세요")가 나왔을 때, **지시받은 동작을 일부러 하지 않고** 그냥 정면을 가만히 보고만 있는다. | A2 | 그래도 PASS가 뜨면 → face mission이 지시된 제스처를 실제로 검증하지 않고 spoof_score만 보는 것. A2 "나쁨" 확정 |
| G2 | 본인이 가만히 정면을 보고 있는 **녹화 영상**을 폰으로 틀어서 카메라에 들이댄다. 얼굴 미션 지시가 나와도 그 영상은 반응하지 않는다(정지 영상이나 다름없음). | A2 (역방향 검증) | PASS가 뜨면 G1과 같은 결론. FAIL이 뜨면 적어도 일부 신호는 라이브 반응을 본다는 뜻 |
| G3 | 손 제스처 미션이 나왔을 때(**"볼 찌르기"가 아닌 다른 미션**, 예: 브이하기/따봉/OK사인/흔들기/밀기 — 중요: 볼 찌르기로 테스트하면 우연히 막혀서 오판하게 됨), **자기 진짜 손**으로 정확히 그 제스처를 취하면서, 동시에 다른 손(또는 거치대)으로 **얼굴 부분만 재생 영상으로 가린다.** | B3/B4 | PASS가 뜨면 → "볼 찌르기" 외 6종 미션에서는 손-얼굴 결합 검증이 없다는 뜻. 우선순위는 "도입"이 아니라 "볼 찌르기에 있는 검증 로직을 나머지 미션에도 확장"으로 재정의 |
| G3-b | "볼 찌르기" 미션이 나왔을 때, 손가락을 볼이 아니라 **얼굴에서 먼 곳(가슴 앞 등)에서 흔들기만** 한다. | B3 (찌르기 미션 자체의 신뢰도) | PASS가 뜨면 → "볼 찌르기"도 사실 위치를 안 본다는 뜻. 이러면 카탈로그에 occlusion성 미션이 있어도 실질적 방어력은 0 |
| G4 | 캡차를 10~20회 반복 시도하면서 나오는 미션 종류(얼굴 지시 문구, 손 제스처 종류, 손가락 번호 조합)를 전부 적어서 unique 개수를 센다. | C5/C6 | 이미 카탈로그상 확인됨(얼굴 10종 + 손 7종+조합) — 이 테스트는 **실제 RNG가 그 풀 전체를 고르게 쓰는지**(특정 미션이 비정상적으로 자주/거의 안 나오는지) 검증용으로만 의미 있음 |
| G5 | 3라운드 진행 중 **카메라를 손으로 가리거나 정지 화면을 계속 띄운 채로 라운드를 넘긴다.** 동시에 개발자도구 Network 탭에서 라운드마다 서버로 보내는 요청의 timestamp/payload가 실제로 바뀌는지 확인. | D7/D8 | 매 라운드 요청 내용이 거의 동일하면 → 라운드가 진짜로 독립적인 재검증이 아니라 같은 입력을 재사용/재채점하는 것일 수 있음 |
| G6 | OBS 등으로 **가상 카메라**를 만들어 미리 준비한 영상을 송출하고, 브라우저 카메라 선택 메뉴에서 그 가상 카메라를 선택해 캡차를 시도한다. | E9/E10 | 정상적으로 캡차가 진행되면 → 인젝션 방어 없음. 카메라 선택 메뉴 자체에서 막히거나 위젯이 경고를 띄우면 → 최소한의 방어는 있음 |
| G7 | 개발자도구 Network 탭에서 `/predict` 또는 `/decide` 호출의 응답 payload를 확인 — `threshold`, `risk_band`, 모델 버전 태그가 찍히는지 본다. | F12/F13 | threshold 값이 0.21 근처면 원본 v1 추정, 그 외 값이면 다른 버전. `risk_band`가 응답에 없으면 캡차 엔진이 ML팀 레퍼런스 `captcha_decision.py`와 다른 자체 로직을 쓰고 있다는 뜻 |

**가장 먼저 시켜볼 테스트**: G1(또는 G2)과 G3 두 개만 해봐도 A2와 B 둘 다 답이 나온다. 이 둘이 지난 분석에서 "결론을 가르는 80%"라고 했던 부분과 정확히 일치한다.

## 답변에 따른 후속 조치 매핑

| 답변 결과 | ML팀이 해야 할 일 |
|---|---|
| G3에서 "볼 찌르기" 외 6종 미션이 손-얼굴 결합 검증 없이 PASS | "볼 찌르기"에 쓰인 위치 검증 로직을 나머지 6종 손동작에도 동일하게 적용(예: 손 bbox가 얼굴/몸통 bbox 근처에 있는지 최소한의 거리 체크라도) 요청 — 완전히 새로 만드는 게 아니라 이미 있는 걸 확장하는 것이라 ROI 높음 |
| G3-b에서 "볼 찌르기"조차 위치를 안 봄이 확인 | 카탈로그상 occlusion 미션이 있어도 무의미 — 이 미션부터 위치 검증 로직을 신규로 요청 |
| A2가 "spoof_score 재탕"으로 확인 | 미션 종류가 풍부한 게 무의미해짐 — 미션별 전용 검증(눈 감김/고개 방향 등 실제 제스처 인식) 도입을 위젯팀에 요청, ML팀이 얼굴 제스처 검출 모듈 개발 지원 검토 |
| F(모델 버전)가 "원본 v1"으로 확인 | 종횡비 버그 수정 모델로 교체를 우선순위 상향 — 단, 9장 결론대로 FRR/replay trade-off는 여전히 남음 |
| D(라운드 독립성)가 "동일 스트림 허용"으로 확인 | 시스템 레벨 risk 누적(`spoof_detected≥2/3`)의 실효성이 낮다고 봐야 함 — 재평가 필요 |
| E(인젝션 방어)가 "없음"으로 확인 | 별도 보안 과제로 분리, 이번 모델 작업의 범위 밖임을 명시 |
