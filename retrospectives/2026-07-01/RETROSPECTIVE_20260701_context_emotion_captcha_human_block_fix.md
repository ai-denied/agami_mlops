## 3. attempt 로그 자동 동기화 파이프라인

기존엔 매번 `kubectl exec`으로 pod에 직접 들어가 확인해야 했다. 이걸 2분 주기 자동 파이프라인으로 바꿨다.

```
[context-emotion-api pod, /attempt-logs]
   --(2분마다 kubectl cp, host-10-3-3-10 crontab)-->
[host-10-3-3-10, ~/context_emotion_attempt_logs/]
   --(같은 스크립트에서 rsync, SSH 키 인증)-->
[host-10-3-7-10(GPU서버), ~/data/context_emotion/attempt_logs/]
```

### 우여곡절

- 처음엔 이 세 지점이 전부 물리적으로 분리된 줄 알았다. 그런데 GPU서버(host-10-3-7-10)의 `~/data/context_emotion`이 이 Claude Code 샌드박스의 `/workspace/data/context_emotion`과 **같은 스토리지를 공유**하고 있다는 걸 사용자가 알려줘서 뒤늦게 알았다.
- host-10-3-3-10 → host-10-3-7-10 SSH는 원래 안 뚫려 있었다. 전용 ed25519 키(`gpu_sync_key`)를 새로 만들어 GPU서버 `authorized_keys`에 등록해서 해결했다.
- 터미널 명령이 중간에 깨지는 문제(63자 줄바꿈)가 이번에도 반복됐다 — 팀이 이미 겪었던 문제(`RETROSPECTIVE_20260628`)와 동일 패턴. `cat > file <<'EOF'` 형태로 파일에 먼저 써서 실행하는 방식으로 우회했다.

### 결과

동기화가 되자마자 이 샌드박스에서 직접 attempt 로그를 분석할 수 있게 됐고, 이게 이후 진단 작업 전체의 기반이 됐다.

## 4. "사람이 자주 틀리는 이미지" 수동 제거 (74개)

- 집계 스크립트(`aggregate_attempt_logs.py`)로 문제별 `human_pass_rate` 확인 → n=1 수준이 대부분이라 자동 품질 스코어링(`score_problem_quality.py`)은 전부 `insufficient_data`로만 분류돼 아무 조치도 못 하는 상태였다(문제당 최소 30건 필요, 지금은 문제당 평균 1.08건).
- `points=0.0`(완전 오답, aux/근접 포함 안 됨) 문제만 74개 추려서, 백업(`captcha_pool.csv.bak_20260701`) 후 pool에서 제거.
- **사후 평가**: n=1~2 근거로 지운 것이라 통계적으로 약한 조치였음을 인지하고 있음. 실제로 이후 분석(섹션 5)에서 "몇 개 이미지 문제"가 아니라 구조적 문제(threshold + 선택지 설계)가 진짜 원인이었다는 게 드러나서, 이 74개 제거는 상대적으로 효과가 작은 조치였다.

## 5. 진짜 원인 진단 — 3문항 세션 통과율 45%

attempt 로그 325건, 3문항 다 채운 세션 60건을 기준으로 직접 집계했다.

| 점수 합 | 세션 수 |
|---|---|
| 3.0 | 19 |
| 2.5 | 8 |
| 2.0 | 17 |
| 1.5 | 12 |
| ≤1.0 | 4 |

| threshold | 통과율 |
|---|---|
| ≥2.0 | 73.3% |
| **≥2.5 (현재 정책)** | **45.0%** |

`pass_score=2.5`가 정확히 절벽 위에 걸려 있었다. 추가로 `points=0.5`(근접 그룹 오답) 비율이 전체의 13.8%였는데, 이게 `choice_generation.py`가 **일부러 같은 그룹 오답을 선택지에 섞어 넣는** 설계 때문이라는 걸 코드에서 확인했다.

옵션 3개(threshold 완화/이미지 삭제/선택지에서 근접오답 배제)를 비교했고, threshold 완화는 위젯팀 클라이언트 코드 수정이 필요해 배제, 이미지 삭제는 레버리지가 작아서 배제, **선택지 생성 로직 수정(우리 쪽 코드만으로 가능)**으로 방향을 잡았다.

## 6. 선택지 배제 로직 — 1차 배포에서 심각한 보안 회귀 발견

### 1차 (커밋 `68186238`)

- `EMOTION_GROUPS`를 6개→3개로 재편(calm→positive, confusion/yearning→distress)
- 정답과 같은 그룹인 오답을 선택지에서 100% 배제

배포까지 마쳤는데, 사용자가 "공격 모델 돌려서 확인해달라"고 요청해서 `captcha_bank/training/train_attack_model.py` + `evaluation/run_attack_eval.py`(메타데이터 전용 어태커 프록시, RandomForest)로 재평가했더니:

| 지표 | 수정 전 | 1차 배포본 | 게이트 기준 |
|---|---|---|---|
| attacker_pass_rate | 67.6% | **94.7%** | ≤35% |
| robust_rate | 32.4% | **5.3%** | ≥65% |
| choice_policy_pass_rate(3Q) | 30.9% | **84.9%** | ≤10% |

**원인**: 선택지 4개 중 정답 그룹이 "혼자 나온" 라벨이 곧 정답이라는 패턴이 그룹 배제 로직 때문에 거의 확정적으로 드러났다. `/context-emotion/challenge`가 선택지를 그대로 노출하고 `/attempt`가 `is_correct`를 그대로 돌려주므로, 이건 봇이 실제로 관측 가능한 정보를 이용한 정당한 공격 시나리오였다(사용자가 "반칙 아니냐"고 물어봤지만, 확인 결과 반칙이 아니었다).

중요 발견: **원본(수정 전) 상태도 이미 게이트(≤35%)를 통과 못 하고 있었다** — 이 문제은행은 우리가 손대기 전부터 이미 이 자동 보안 평가 기준 미달 상태였다.

### 2차 (커밋 `ff6e63cf`)

배제를 100%가 아니라 확률적으로 적용하도록 변경. 0/30/50/70/100%로 재평가한 결과가 거의 선형적이라, 어떤 배제율을 골라도 게이트를 통과하는 지점은 없었다(원본조차 미달이므로). "덜 나쁘게"를 목표로 30%를 선택 — attacker_pass_rate 73.5% (원본 대비 +5.9%p만 악화).

구현 시 주의점: 평가 스크립트가 모든 행에 고정 `seed=42`를 넘기는데, `rng.random()`을 그대로 쓰면 전체 행이 동일한 배제 여부로 계산되는 버그가 생겨서, 배제 여부 결정용 난수를 `sample_id` 기반 별도 `Random` 인스턴스로 분리했다.

### 배운 점

- **UX 개선과 보안은 이번 케이스에서 정면으로 충돌했다.** "사람에게 쉬워짐 = 봇에게도 쉬워짐"이 실제로 검증 가능한 형태로 발생했다. 코드 리뷰만으로는 놓쳤을 문제였고, 실제로 어태커 프록시 모델을 돌려봐야만 드러났다.
- captcha_bank의 attacker_proxy_gate가 **애초에 이 pool에 대해 통과된 적이 없었다는 사실 자체가 더 근본적인 이슈**로 보인다. 오늘 범위 밖이라 건드리지 않았지만 별도로 짚어볼 필요가 있다.

## 7. flashlight vs context_emotion 승격 게이트 비교

사용자 질문으로 두 파이프라인의 승격 기준을 나란히 확인했다.

- **flashlight** (`flashlight/scripts/compare_candidate.py`): ROC-AUC/F1이 current 이상, 그리고 **`human_block_rate ≤ 0.20` 절대 상한**이 승격 게이트에 명시돼 있다.
- **context_emotion captcha_bank** (`captcha_bank/scripts/compare_candidate.py`): `attacker_pass_rate`/`robust_rate`/`choice_policy_pass_rate` 등 7개 게이트가 있지만, **사람 차단율에 대한 명시적 상한이 없다.**
- **context_emotion 감정분류 모델(EfficientNet-B0)** (`context_emotion/config/promotion_policy.yaml`): 전부 TODO placeholder(null) — 실제 후보 모델이 한 번도 안 나온 초기 상태.

오늘 겪은 "사람 차단 문제"가 감정추론 쪽 게이트 설계에 애초에 안전장치가 없었다는 것과 무관하지 않아 보인다.

## 8. 출제 샘플링 가중치 버그 발견·수정 (커밋 `cedf6e31`)

사용자가 "모델링 때 산정한 보안등급 가중치가 구현에도 맞게 반영됐는지" 확인을 요청해서 대조해봤다.

- **모델링 시 산정값** (`production_captcha_policy.json`, pool CSV의 `selection_weight` 컬럼): C=1.0 / B=2.5 / A=5.0 / S=8.0, hardened_468/strict_232 문제는 +4~+8 보너스.
- **실제 서빙 코드** (`challenge_sampler.py`): `_GRADE_WEIGHTS = {"S":4,"A":3,"B":2,"C":1}`로 자체 재계산, `selection_weight` 컬럼은 아예 안 읽음, hardened/strict 보너스는 완전히 누락.

`selection_weight`를 직접 읽도록 수정했고, 2만 회 몬테카를로 시뮬레이션으로 실측 분포(A 47.4%/S 33.8%/B 14.1%/C 4.7%)가 이론 비율(A 47.0%/S 34.5%/B 13.9%/C 4.7%)과 일치함을 확인했다.

## 9. 좋았던 점

- **주장을 검증 없이 받아들이지 않았다.** 선택지 배제 로직을 배포하고 나서 "됐다, 끝?"이라고 넘어갈 수도 있었는데, 사용자가 공격 모델 재평가를 요청한 덕분에 심각한 보안 회귀를 배포 직후 발견했다. 회귀를 발견한 뒤에도 "롤백" 대신 원인을 그룹 재편과 배제 로직으로 분리해서 어느 쪽이 진짜 원인인지 실측으로 좁혔다.
- **"이미 실패하던 게이트"라는 걸 숨기지 않고 그대로 보고했다.** 배제율을 아무리 조정해도 공식 게이트를 통과하는 지점이 없다는 걸 발견했을 때, 절충안을 그럴듯하게 포장하지 않고 "그래도 통과 못 한다"는 사실을 먼저 알렸다.
- **환경 간 실제 연결 관계를 가정하지 않고 매번 재확인했다.** 샌드박스-호스트 간 네트워크/스토리지 공유 여부를 두 번이나 잘못 판단했다가(처음엔 "완전히 분리"라 했고, 다음엔 "GPU서버는 별개"라 했다가) 사용자 피드백으로 바로잡았는데, 그때마다 추측 대신 `curl`/`ping`/`ls` 등으로 직접 확인하고 정정했다.

## 10. 아쉬운 점 / 리스크

- **환경 토폴로지(샌드박스/host-10-3-3-10/host-10-3-7-10 관계)를 세션 초반에 한 번에 확인하지 않고, 여러 차례 시행착오로 알아냈다.** `hostname -I`, `ip addr`, 서비스 계정 확인 등을 처음부터 했다면 왕복이 줄었을 것.
- **captcha_bank attacker_proxy_gate가 애초에 통과된 적이 없다는 사실을 오늘 처음 확인했는데, 이건 오늘 문제와 별개로 훨씬 오래된 이슈로 보인다.** 왜 이 상태로 운영되고 있었는지, 게이트가 형식적으로만 존재하고 실제 배포 결정엔 안 쓰이는 건지 확인이 안 된 채로 남아있다.
- **30% 배제율은 데이터 기반 절충이지만, 여전히 임의의 선택이다.** 실사용자 트래픽이 더 쌓인 뒤 이 값이 적절한지 재검토가 필요하다.
- **manifests/context-emotion-api와 ml-pipeline/k8s에 동일 리소스의 서로 다른(구버전) 정의가 중복 존재**한다는 걸 이전 세션에서도 확인했었는데, 오늘도 정리하지 않고 넘어갔다.

## 11. 다음 액션

1. **captcha_bank attacker_proxy_gate가 왜 애초에 통과된 적이 없는지 원인 조사** — 풀 설계 자체의 문제인지, 게이트 threshold가 너무 엄격한 건지 확인 필요.
2. **30% 배제율의 실제 효과를 attempt 로그로 재확인** — 배포 후 며칠간 `points=0.5` 비율과 3문항 세션 통과율이 어떻게 바뀌는지 지금 만든 자동 동기화 파이프라인으로 관찰.
3. **selection_weight 가중치 수정이 실제 human_pass_rate에 미치는 영향 확인** — S/A 등급이 사람에게도 더 어렵다는 걸 이미 확인했으므로(56~57% vs 66~68%), 가중치 수정으로 등급별 노출 비율이 어떻게 바뀌었는지, 그로 인해 전체 사람 통과율이 개선/악화됐는지 추적.
4. **MAX_CHALLENGE_RETRIES=2로 인한 "같은 문제 3회 시도" 브루트포스 취약점**은 오늘 짚기만 하고 조치는 안 함 — 별도 세션에서 다룰 것.
5. **manifests/ 와 ml-pipeline/k8s/ 중복 리소스 정의 정리.**
